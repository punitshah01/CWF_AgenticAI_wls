1. Executive Overview

This test plan characterizes Clearwater Forest (CWF) platform performance under Agentic AI workloads — a new class of compute tasks where LLM-powered autonomous agents solve complex multi-step problems by interacting with environments (code repos, web browsers, desktops, APIs).

Why CWF for Agentic Workloads? CWF's high core-count E-core architecture is uniquely suited for agentic AI because: (1) LLM inference is memory-bandwidth bound — more cores spread BW pressure, (2) environment execution (Docker, VMs, browsers) benefits from massive parallelism, (3) multi-instance agent deployment scales naturally with core count, (4) efficiency (perf/watt) matters for always-on agent services.

4. Deployment Architecture on CWF

Two-Domain Compute Model Agentic workloads have two distinct compute domains that must be characterized independently:

Domain 1 — LLM Inference (The "Brain"): Memory-BW bound, vectorized matmul, large working set, benefits from AVX-VNNI
Domain 2 — Environment Execution (The "Body"): I/O heavy, process spawning, Docker/KVM orchestration, bursty, benefits from core count

Inference Engine Options for CWF

Engine Backend Intel Optimization Best For Quantization
vLLM OpenVINO / PyTorch OpenVINO backend auto-detects Intel features Multi-instance serving, batched requests INT8, INT4 (GPTQ/AWQ)
llama.cpp GGML/GGUF Explicit AVX-VNNI codepaths Single-instance, low memory, fast startup Q4_0, Q4_K_M, Q8_0
OpenVINO GenAI OpenVINO IR Native Intel graph optimization Production deployment, max throughput INT8, INT4 (NNCF)
IPEX-LLM Intel Extension for PyTorch Custom kernels for Xeon Research, flexibility INT8, INT4, FP16

7.3 EMON Collection Plan

EMON Strategy for Agentic Workloads Agentic workloads are bursty — long idle waits for LLM response followed by burst environment activity. Use time-series EMON (not just summary) to capture phase behavior. Recommended: 100ms sample interval, socket-view + core-view for placement analysis.
Phase EMON Views Key Events

8. Execution Phases & Timeline

Phase 0: Environment Validation (Day 1)

Goal: Confirm CWF system is ready — Docker, KVM, Python 3.11, storage, memory.

  • Verify KVM: egrep -c '(vmx|svm)' /proc/cpuinfo
  • Docker: run hello-world container
  • Storage: confirm 200+ GB free
  • Memory: confirm 128+ GB available
  • Install vLLM / llama.cpp, download model weights
  • Validate inference: run simple prompt, measure tokens/sec baseline

Phase 1: AppWorld + T-Bench (Days 2–3)

Goal: Lightweight benchmarks first — fastest iteration, validate full pipeline.

  • AppWorld: install, verify, run on dev set (quick feedback)
  • T-Bench: setup mock API server, run full suite
  • Measure: inference tokens/sec, task completion rate, end-to-end latency
  • Core scaling: sweep inference cores 16→32→64→128
  • Multi-instance: 1→2→4 AppWorld agents in parallel

Phase 2: SWE-bench (Days 4–6)

Goal: Software engineering benchmark — Docker-based, moderate complexity.

  • Pull Docker images for SWE-bench Lite (300 tasks)
  • Run gold-patch validation first
  • Agent evaluation: SWE-agent + local LLM
  • Core scaling: vary max_workers (1,2,4,8,16)
  • EMON collection during peak load
  • Measure: resolve_rate, time/task, storage I/O

Phase 3: WebArena (Days 7–9)

Goal: Full web services deployment — tests multi-container orchestration.

  • Deploy 5 web services (Shopping, GitLab, Reddit, Wikipedia, Map)
  • Validate each service is accessible
  • Run 812-task evaluation
  • Monitor: Docker resource usage, network I/O, memory pressure
  • Placement experiment: services co-located vs spread across NUMA

Phase 4: OSWorld (Days 10–12)

Goal: Desktop VM benchmark — heaviest workload, tests KVM + vision model.

  • Verify Docker+KVM nested virtualization
  • Download Ubuntu VM images
  • Run with num_envs scaling: 1→2→4→8→10
  • Requires multimodal model (screenshot) — may need larger model
  • Measure: per-domain success rate, VM boot time, screenshot capture latency

Phase 5: Full Scaling Characterization (Days 13–15)

Goal: Combined scaling study across all benchmarks with EMON.

  • Best-performing config from each phase
  • Full EMON collection (socket + core + uncore views)
  • Power characterization: RAPL at each scaling point
  • Topology experiments: compact vs spread placement
  • Generate final comparison report

9. Risks & Mitigations

Risk Severity Impact Mitigation
LLM inference too slow on E-cores (low IPC) High Tasks timeout, low success rate Use INT4 quantization; increase decode batch; try smaller models first
Insufficient memory for 70B model + environments Medium OOM kills, failed tasks Start with 8B model; monitor RSS; use memory-mapped GGUF
OSWorld KVM not supported in nested virt Medium Cannot run OSWorld benchmark Verify KVM before Phase 4; fall back to VMware if needed
Docker image pull bandwidth (120+ GB for SWE-bench) Low Slow setup, wasted time Pre-pull images overnight; use local registry cache
Benchmark non-determinism confounds scaling study Medium Can't distinguish platform effect from LLM variance Run 3+ repetitions per config; report mean + stddev; fix random seed where possible
WebArena services consume too many resources Low Starves inference cores Pin service containers to dedicated cpuset; monitor with cgroups
Model quality too low for meaningful benchmark scores High 0% pass rate makes scaling study meaningless Validate with known-good model (API) first to establish ceiling; use 32B+ for SWE-bench
Thermal throttling at full core utilization Medium Performance drops during sustained load Monitor freq + temp; ensure adequate cooling; run steady-state warmup
Critical Success Factor The #1 risk is model quality vs. inference speed tradeoff. A fast but inaccurate small model gives meaningless benchmark scores. Recommendation: Start with 8B for pipeline validation, then use 32B+ for actual characterization data. Consider running a cloud-API baseline (GPT-4o) to establish the "ceiling" score before local inference testing.
